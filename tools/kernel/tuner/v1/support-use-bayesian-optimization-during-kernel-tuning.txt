This is a design doc of using Bayesian optimization for tuning kernels.
Please refer to how the current kernel tuner works through the README.md file. But overall how the tuning works is:
1. program entry point is the kernel_tuner_runner.py, this can be invoked via command line like 
   python -m tools.kernel.tuner.v1.kernel_tuner_runner --run_locally --kernel_tuner_name=batched_rpa_kernel_tuner --case_set_desc=batched_rpa_kernel_tuning_setup --case_set_id=batched_rpa_2 --run_id=0 --tpu_version=tpu7x --tpu_cores=2
2. After the invoked, it will go through 2 steps, 1st step is generating the tuning cases(a tuningkey and tunableparams pair) and create buckets(a set of tuning cases).
 and 2nd step is running each tuning case and measure the latency. currently we sweep through all the tuning cases. With select the cases with the same tuningkey and has the minimal latency to be the best tunableparams for that tuningkey. 

The kernel_tuner_runner is the entrypoint to invoke the kernel tuner defined in the kernel_tuner_runner's KERNEL_TUNER_REGISTRY. each kernel tuner is a subclass of the common/kernel_tuner_base.py. 
The kernel_tuner_base defines the common infra for different kernel tuners like storage management, latency measurement, error handling etc. the subclass override several abstract member functions to generate kernel specific tunable cases,
setup the kernel input according to the tuningkey, kernel invocation for each kernel case. 

Currently the tuning case bucket can be created with fixed size, for example, each bucket can have 500 tuning cases. this is for balacing the load across multiple machines when we need to sweep hundreds of thousands of tuning cases. now we want to 
 Sweeping all the tuning cases is not an efficient way to do kernel tuning. for example, after somepoint, keep increasing a tunable param will only make the latency larger so there is no point to sweep beyond that. The bayesian optimization technique
 seems to be a good optimization tool/methodology here. When using bayesian optimization, for the optimization target, it's based on the tuningkey. so to make it easier, the bucket should be for one tuningkey. all the tunable params combination 
 for the same tuningkey should be put into the one bucket. and one bucket should only contains all the tuning cases with the same tuningkey. 

I want you to use optuna, which is currently the industry standard for this exact problem. some code snippets:
import optuna

# 1. This is your "Black Box" function
def run_my_blackbox(learning_rate, depth, strategy):
    """
    This function represents your actual system.
    It could be training a model, running a physics simulation, 
    or executing a trading backtest.
    """
    # Simulate a result based on the inputs
    if strategy == "aggressive":
        score = (learning_rate * 100) - (depth ** 2)
    else:
        score = (learning_rate * 50) - (depth * 1.5)
        
    return score

# 2. Define the Objective for the Optimizer
def objective(trial):
    # Define your search space using the 'trial' object
    
    # Continuous variable between 0.001 and 0.1 (log scale is often best for small floats)
    lr = trial.suggest_float("learning_rate", 1e-3, 1e-1, log=True)
    
    # Integer variable between 1 and 10
    depth = trial.suggest_int("depth", 1, 10)
    
    # Categorical variable 
    strategy = trial.suggest_categorical("strategy", ["conservative", "aggressive"])
    
    # Run the black box with the suggested parameters
    result = run_my_blackbox(lr, depth, strategy)
    
    # Return the target value you want to optimize
    return result

# 3. Run the Optimization
if __name__ == "__main__":
    # Create a study (set direction to "maximize" or "minimize" your target)
    study = optuna.create_study(direction="maximize")
    
    # Run 50 experiments (trials)
    study.optimize(objective, n_trials=50)
    
    # Output the results
    print("\nOptimization Complete!")
    print(f"Best Target Value: {study.best_value}")
    print(f"Optimal Parameters: {study.best_params}")
